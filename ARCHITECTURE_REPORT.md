## 🔄 4. Data Flow Patterns

The application follows a **"One-Way Clinical Flow"** to ensure data integrity.

### 1. Data Fetching (Server-to-Client)
1. **Server Components** fetch static or dynamic data directly from Supabase via `createServerClient`.
2. Data is passed as props to **Client Components** for interactivity.
3. This pattern ensures high performance by reducing the initial "Main Thread" blocking time.

### 2. Data Mutation (Optimistic Success)
1. User actions (e.g., booking a slot, adding notes) trigger a **React Query Mutation**.
2. **Optimistic UI Update**: The UI updates instantly (e.g., "Booking Confirmed" state) while the background sync occurs.
3. If a failure occurs, the state is rolled back with an **empathetic microcopy notification** ("Connection Before Correction").

### 3. Safety Check (Middleware & RLS)
1. **Next.js Middleware** interceptor checks the JWT in the cookies.
2. If authenticated, the request proceeds to the route.
3. **PostgreSQL RLS** ensures that even if a frontend check is bypassed, a therapist can *only* see their own bookings and a parent can *only* see their children's data.

## 🔒 5. Security & Dev Protocol

- **Zero "Hard Failures"**: UI errors are handled with supportive text, never raw system errors.
- **TypeScript Absolute**: No `any` types allowed. Every entity (Student, Therapist, Payout) is strictly interface-bound.
- **Scale Strategy**: Modular monorepo design allows the system to easily grow from 10 users to 1,000+ without structural refactoring.
- **RLS Sovereignty**: Security policies are kept as Migration SQL files (e.g., `2026xxxx_RLS_update`) to maintain an institutional development history.
